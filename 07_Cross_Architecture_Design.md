# 07. Cross-Architecture Design — "다른 모델로 풀어봐" 사고

> **교수님 직격타 질문**:
> *"이미지를 CNN이 아니라 RNN으로 학습하려면 어떻게 설계해야 하고, 무엇이 다르냐?"*
>
> 이 챕터는 그런 질문에 **표준 답안 패턴**을 만드는 게 목적.

---

## 0. 큰 그림 — "왜 이런 질문을 하나?"

모델 = **inductive bias의 묶음**.
- CNN: 국소성 + 평행이동 등변성 + 계층적 합성.
- RNN: 시간적 순서 + 시간 축 가중치 공유.
- Transformer: 거의 없음 (위치 인코딩으로 약하게).
- FNN: 거의 없음 (모든 입력 동등).

데이터 = **본래의 구조**가 있다 (이미지=2D 격자, 텍스트=1D 시퀀스, 그래프=노드+엣지).

→ "**데이터의 구조와 모델의 bias가 맞으면 효율적**, 어긋나면 비효율적이거나 정보를 잃는다."

따라서 cross-architecture 설계는 항상 다음 3개를 답해야 한다:
1. **데이터를 어떻게 변환?** (입력 형식 맞추기)
2. **무엇을 잃는가?** (잃는 inductive bias)
3. **무엇을 얻는가?** (왜 그래도 시도할 만한가, 또는 왜 합리적이지 않은가)

---

## 1. 이미지를 RNN으로 학습 (★ 단골)

### 1.1 어떻게 설계?

이미지(H×W×C)는 2D인데 RNN은 1D. → **시퀀스로 펼친다**.

| 방식 | 어떻게 | 특징 |
|---|---|---|
| Raster scan | 좌→우, 위→아래 | 가장 단순, 행 끝에서 의존성 끊김 |
| Hilbert curve | 공간국소성 보존하는 곡선 | 인접 픽셀이 시퀀스에서도 인접 |
| Row-LSTM / Diagonal BiLSTM | 한 행 또는 대각선 단위 처리 | PixelRNN의 변종 |
| Patch sequence | 16×16 patch를 시퀀스로 | ViT 식 (Transformer지만 같은 사고) |

### 1.2 대표 사례
- **PixelRNN** (DeepMind 2016): 픽셀 시퀀스의 자기회귀 모델. 한 픽셀씩 생성.
- 이전 픽셀로 다음 픽셀의 분포 예측 → **이미지 생성**에 강.

### 1.3 무엇을 잃나? (★)
- **2D 공간 inductive bias 상실**: CNN의 weight sharing이 2D에서 작동하던 걸, RNN은 시간 축으로만 sharing.
- **장거리 픽셀 간 의존성 약함**: 224×224 → 50,176 step. RNN은 그렇게 멀리 정보 전달 어려움. (LSTM도)
- **인접성 왜곡**: raster scan에서 (0, 223)과 (1, 0)은 시퀀스상 인접하지만 공간상 멀다.
- **병렬화 불가**: CNN은 모든 위치를 동시에. RNN은 순차.

### 1.4 무엇을 얻나?
- **명시적 자기회귀 분포**: $P(x) = \prod P(x_i | x_{<i})$ — 픽셀 생성에 자연스럽다.
- **가변 해상도** (이론상): RNN은 길이 무관.

### 1.5 결론
- **분류/인식에는 매우 비효율**: CNN이 압도. 같은 일을 5050× 느리게.
- **생성에는 의미 있었음**: PixelRNN/PixelCNN 시대. 지금은 Diffusion이 표준.

### 1.6 예상 답변 골격 (시험·면접용)
> "이미지를 RNN으로 풀려면 픽셀을 raster scan(또는 Hilbert) 식으로 1D 시퀀스로 펼치고, 각 시점에 픽셀 또는 patch를 입력으로 LSTM/GRU에 넣는 방식 (PixelRNN). 다만 CNN이 가진 **국소성·평행이동 등변성**이라는 강력한 2D inductive bias를 잃기 때문에 같은 성능을 내려면 **훨씬 더 많은 파라미터와 데이터, 시간**이 필요하다. 224×224라면 5만 step 이상의 시퀀스가 되어 LSTM의 long-range 한계에 부딪히고, 병렬화도 불가능해 학습 속도가 매우 느리다. **분류엔 부적절하고**, 자기회귀 생성에는 합리적이다."

## 2. 텍스트를 CNN으로

### 2.1 어떻게?
- 단어를 embedding (V → d 차원).
- 시퀀스 = (T × d) → **1D conv**를 d 채널 위에 슬라이딩.
- 커널 크기 k = 2,3,4 (= n-gram).
- max-over-time pooling으로 길이 무관 고정 벡터.
- → **TextCNN** (Yoon Kim, 2014).

### 2.2 무엇을 얻나?
- **병렬화** (RNN 대비 매우 빠름).
- **국소 패턴** (n-gram)을 잘 잡음 — 감성 분석에 효과적.
- **단순/강력한 baseline**.

### 2.3 무엇을 잃나?
- **장거리 의존성**: RF가 깊이로 제한. "I went to ... store" 같은 멀리 떨어진 의존 약함.
- **순서 정보 일부**: max pooling이 위치를 흐림.

### 2.4 언제 합리적?
- 짧은 텍스트 분류 (감성, 토픽).
- 빠른 inference 필요.
- 문법보다 키워드성이 중요한 task.

### 2.5 RNN/Transformer와 비교
| | TextCNN | LSTM | Transformer |
|---|---|---|---|
| 장거리 | 약함 | 보통 | 강함 |
| 병렬 | 가능 | 불가 | 가능 |
| 데이터 효율 | 좋음 | 보통 | 많이 필요 |
| 짧은 분류 | 강력 | OK | overkill 가능 |

## 3. 시계열을 FNN으로

### 3.1 어떻게?
- **Sliding window**: 최근 N개 시점을 평탄화해 입력 벡터(N×d)로.
- 출력: 다음 시점.

### 3.2 무엇을 잃나?
- **가변 길이 처리 불가** (N 고정).
- **시간 축 가중치 sharing 없음**: 시점 t의 패턴과 t-5의 같은 패턴을 따로 학습.
- **N 너머 정보** 완전 차단.
- N을 키우면 파라미터 폭발.

### 3.3 무엇을 얻나?
- 단순.
- 단기 예측에 의외로 강함 (baseline).
- 학습 빠름.

### 3.4 언제?
- 짧은 의존성으로 충분한 문제 (예: 다음 시점 가격, 단기 수요).
- 시계열에 강한 외생 변수가 dominate하는 문제.

생각해보라: **왜 1D CNN(temporal conv)가 FNN보다 나은가?** → 시간 축 sharing이 있다. 같은 패턴을 한 번 학습하면 모든 시점에 적용. WaveNet, TCN의 동기.

## 4. 시퀀스를 Transformer로 (RNN 대신)

### 4.1 어떻게?
- 입력 시퀀스 (T × d)에 **self-attention**.
- 위치 정보는 **positional encoding**으로 주입 (sin/cos 또는 학습).
- Multi-head attention + FFN을 stack.

### 4.2 RNN 대비 다른 점
| | RNN | Transformer |
|---|---|---|
| 시간 축 처리 | 순차 (h_t는 h_{t-1} 필요) | 동시 (모든 시점 attention) |
| Long-range | 거듭제곱 효과로 약함 | 거리 무관 직접 연결 |
| 병렬화 | ✗ (학습 시 teacher forcing 제외) | ✓ |
| Inductive bias | 시간 순서 강함 | 거의 없음 (PE만) |
| 메모리 | O(T·d) | O(T²·d) ★ |
| 데이터 요구 | 적당 | 많음 |
| 위치 정보 | 자동 (sequential) | 명시적 PE 필요 |

### 4.3 무엇을 얻나?
- **장거리 의존성 직접**: 토큰 1과 토큰 1000을 한 layer에서 연결.
- **병렬 학습**: GPU 활용 극대화 → 큰 모델 가능.
- **scaling이 잘 됨**: 데이터·파라미터·연산을 늘릴수록 성능 향상.

### 4.4 무엇을 잃나?
- **순서 정보의 자연스러움**: PE에 의존. 잘못 설계하면 장거리에서 무너짐.
- **메모리 O(T²)**: 긴 시퀀스에 부담. → Sparse/Linear attention, FlashAttention 등.
- **데이터·연산이 많이 필요** (작은 데이터에선 RNN/CNN이 나을 수도).

### 4.5 결론
"NLP가 RNN→Transformer로 간 이유 = **병렬화 + long-range + scaling**. 단 데이터·연산 비용은 더 든다."

## 5. 그래프 데이터를 CNN으로?

### 5.1 그래프의 특성
- 노드 순서 임의.
- 인접 관계가 격자가 아니라 임의 엣지.
- 노드별 차수(이웃 수) 가변.

### 5.2 CNN 직접 적용의 문제
- CNN의 conv는 **격자**와 **고정된 인접성**을 가정.
- 노드 순서를 정해 격자에 매핑하면 → 순서에 따라 결과 바뀜 (**순서 등변성 위배**).
- 노드 차수가 다르면 커널 크기 통일 불가.

### 5.3 해결: GCN (Graph Conv Network)
- "이웃 노드의 표현을 평균/가중합" → 인접 행렬 A를 사용.
- $H^{(l+1)} = \sigma(\hat{A} H^{(l)} W^{(l)})$
- 노드 순서 무관, 차수 무관.

### 5.4 교훈
- **데이터 구조에 맞는 inductive bias가 따로 필요** — CNN을 강제로 적용하지 말 것.
- 격자가 아닌 데이터엔 격자 가정의 모델이 맞지 않음.

## 6. 표(tabular) 데이터를 딥러닝으로?

### 6.1 표 데이터의 특성
- 컬럼(feature) 순서 임의.
- 컬럼 간 의미는 강하지만 위치적 의존 없음.
- feature scale·type이 매우 이질적.

### 6.2 RNN/CNN을 강제 적용하면?
- RNN: 컬럼을 시퀀스로 → "컬럼 순서"가 의미 없으니 학습 신호 부정확.
- CNN: 격자 가정 위배.

### 6.3 합리적 옵션
- **MLP** (가장 단순) + 적절한 normalization·embedding.
- **TabTransformer / FT-Transformer**: feature를 embedding한 후 self-attention.
- **GBM** (XGBoost, LightGBM, CatBoost): 표 데이터의 강자. 딥러닝과 거의 동급 또는 우위.

### 6.4 결론
- 표 데이터는 **딥러닝이 항상 우위가 아님**. 트리 모델이 표준.
- 딥러닝을 쓰려면 컬럼 순서 무관한 구조(attention) 필요.

## 7. 동영상은 어떻게? (시간+공간)

### 7.1 옵션 비교

| 방식 | 구조 | 장점 | 단점 |
|---|---|---|---|
| **2D CNN per frame + LSTM** | 프레임마다 CNN, 시간 축은 LSTM | 단순, 잘 작동 | 시간·공간 분리 |
| **3D CNN** | conv를 H×W×T로 | 시공간 동시 | 파라미터·연산 많음 |
| **(2+1)D CNN** | 2D + 1D 분리 | 효율, 잘 작동 | 약간 표현력 ↓ |
| **Video Transformer** | patch×time을 attention | 강력 | 연산·데이터 多 |

### 7.2 사고 패턴
- **시간/공간 inductive bias 둘 다 필요** → 두 축 어디에 어떻게 분배?
- 짧은 클립(액션 인식): 3D CNN, (2+1)D.
- 긴 영상: 2D CNN + LSTM/Transformer.

## 8. 음성/오디오는?

### 8.1 입력 형식
- **시간 도메인 raw waveform**: 1D 시퀀스 (16,000~48,000 sample/s).
- **주파수 도메인 (spectrogram)**: 2D (시간 × 주파수). → CNN 적용 가능.

### 8.2 모델 매칭
| 입력 | 합리적 모델 |
|---|---|
| Raw waveform | 1D CNN (WaveNet), Transformer |
| Spectrogram | 2D CNN (음성 인식), CNN+RNN |
| Long clip | Transformer (Whisper 식) |

### 8.3 핵심 통찰
- 음성을 spectrogram으로 변환 = "오디오를 이미지처럼 본다" → CNN의 모든 기법 적용 가능.
- **데이터 표현 변환**이 모델 선택을 바꾼다 — 핵심 사고 패턴.

## 9. 일반 원칙 — "Cross-Architecture 설계 사고"

1. **데이터의 본래 구조 파악** (격자? 시퀀스? 집합? 그래프? 표?).
2. **모델의 inductive bias 매칭**.
3. 어긋나면 **데이터 변환** or **모델 변환** or **둘 다**.
4. 변환 시 **잃는 정보**와 **얻는 능력**을 명시적으로 비교.
5. **실용적 결론**: 데이터 양·연산 예산·요구 성능을 고려해 trade-off.

## 10. 표로 보는 "데이터 ↔ 모델 매칭"

| 데이터 | 자연스러운 모델 | 강제 적용 시 잃는 것 |
|---|---|---|
| 이미지 | CNN | 2D 공간 bias (RNN), 위치 정보 (FNN) |
| 텍스트/음성 | RNN/Transformer | 순서 정보 (FNN), 장거리 (CNN) |
| 시계열 | RNN/Transformer/TCN | 시간 sharing (FNN), 가변 길이 (FNN) |
| 그래프 | GNN | 비격자 인접성 (CNN), 순서 무관 (RNN) |
| 표 | MLP/GBM/TabTransformer | 적절 (트리가 강자) |
| 동영상 | 3D CNN / 2D+LSTM / Video Transformer | 시간 또는 공간 (한 쪽만 모델링 시) |
| 오디오(raw) | 1D CNN / Transformer | 시간 sharing (FNN) |
| 오디오(spec) | 2D CNN | 동상 |

## 11. "생각해보라" 확장 질문

1. **시계열을 Transformer로 풀 때 RNN보다 항상 나은가?** → 데이터 적으면 RNN/TCN이 나을 수 있다 (Transformer는 데이터 효율 낮음).
2. **이미지를 시퀀스로 펼칠 때 raster vs Hilbert curve, 어느 쪽이 나은가?** → Hilbert가 공간 국소성 보존 → 인접 픽셀이 시퀀스에서도 가깝다 → 학습 신호 좋다.
3. **TextCNN이 LSTM보다 빠른 이유는 무엇이고, 어떤 task에서 LSTM에 밀리나?** → 병렬화 가능. 장거리 의존성 task(긴 문서 이해)에서 밀림.
4. **표 데이터를 self-attention으로 풀면 무엇이 좋은가?** → 컬럼 간 임의의 상호작용을 데이터로 학습. 단 데이터가 충분해야.
5. **그래프 데이터에 RNN을 적용할 수 있나?** → 그래프를 노드 시퀀스로 펼쳐야 하는데 순서가 임의 → 부적절. random walk 기반 시퀀스화는 가능하나 정보 손실.
6. **Image classification을 FNN으로 한다면? 왜 못 쓰나?** → 위치 이동에 약함, 파라미터 폭발. CIFAR-10에서도 CNN의 절반 정확도도 어려움.

## 12. 한 줄 요약

- 모든 데이터를 모든 모델로 **적용은 가능**, 단 효율과 손실이 다르다.
- 핵심 = "**inductive bias 매칭** 사고".
- 변환 시 항상 "**잃는 것 / 얻는 것 / 합리성**"을 답할 수 있어야 한다.
- 교수님 단골 질문 = 표준 답안 = "이렇게 설계 + 잃는 것 + 그래도 하는 이유 / 안 하는 이유".
